# Hybrid FAQ Retrieval Benchmark

Воспроизводимый стенд для сравнения вариантов поиска по FAQ-корпусу: поиск BM25, векторный поиск, объединение сигналов и переранжирование.

## Назначение

По пользовательскому вопросу находить наиболее релевантный ответ в корпусе FAQ-документов и сравнивать качество разных стратегий поиска в единых условиях.

## Архитектура

```text
FAQ-корпус
    │
    ▼
Подготовка документов и запросов
    │
    ├─ поиск BM25
    ├─ векторный поиск
    └─ объединение сигналов
            │
            ▼
      Переранжирование
            │
            ▼
      Расчёт метрик качества
            │
            └─ API поиска
```

## Что реализовано

- локальный корпус FAQ и набор запросов;
- несколько режимов поиска в одном стенде;
- единый сценарий оценки;
- сохранение сводных результатов запуска;
- API для локальной проверки качества поиска.

## Структура репозитория

- `configs/` — конфигурации экспериментов;
- `data/demo_faq/` — демонстрационный корпус и эталонные соответствия;
- `examples/` — примеры запросов;
- `reports/` — отчёты запусков;
- `scripts/` — сценарии запуска;
- `src/faq_bench/` — основной код;
- `tests/` — тесты;
- `Dockerfile`, `Makefile` — воспроизводимый запуск.

## Оценка качества

Основные метрики:

- `Recall@k`
- `MRR@k`
- `nDCG@k`
- `HitRate@k`
- задержка ответа: `latency_ms_p50`, `latency_ms_p95`

Сравнение режимов:

- поиск BM25;
- векторный поиск;
- объединение сигналов;
- объединение сигналов с переранжированием.

## Результаты

Демонстрационный корпус: 6 запросов, 8 документов. Метрики при `k = 5` из локального запуска `python scripts/run_benchmark.py` с конфигурациями из репозитория.

`scripts/run_benchmark.py` перезаписывает `reports/latest_summary.json` (и связанные детали) при каждом запуске — в таблице ниже зафиксированы значения с одной машины; для своих чисел выполните три запуска с `configs/bm25.yaml`, `dense.yaml`, `hybrid.yaml` и смотрите локальный JSON после каждого. Сохранить снимки по всем трём режимам в git одновременно нельзя: в `.gitignore` указаны `reports/*.json` и `reports/*.md` (кроме `.gitkeep`). `scripts/export_report.py` читает существующий `reports/latest_summary.json` и создаёт `reports/latest_summary.md`; оба файла появляются только когда вы запускаете эти скрипты и по умолчанию не коммитятся.

| Режим (config) | Recall@5 | MRR@5 | nDCG@5 | HitRate@5 | latency_ms_p50 | latency_ms_p95 |
|----------------|----------|-------|--------|-----------|----------------|----------------|
| BM25 (`configs/bm25.yaml`) | 1.0 | 1.0 | 1.0 | 1.0 | 0.037 | 0.063 |
| Векторный поиск (`configs/dense.yaml`) | 1.0 | 1.0 | 1.0 | 1.0 | 12.47 | 15.13 |
| Объединение + переранжирование (`configs/hybrid.yaml`) | 1.0 | 1.0 | 1.0 | 1.0 | 62.41 | 88.28 |

На малом корпусе метрики часто достигают 1.0; для выводов о качестве на реальных данных требуется свой корпус и отдельный запуск бенчмарка.

## Запуск

```bash
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

python scripts/run_benchmark.py --config configs/bm25.yaml
python scripts/run_benchmark.py --config configs/dense.yaml
python scripts/run_benchmark.py --config configs/hybrid.yaml
```

## Ограничения

- эффект векторного поиска зависит от выбранной модели представления текста;
- эффект переранжирования зависит от качества исходного набора кандидатов;
- для переноса в промышленный контур нужен корпус большего объёма и журналирование качества на реальных запросах.
